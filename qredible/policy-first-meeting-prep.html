01 Executive Summary What "policy-first" means ▶

Policy-first means: versioned JSON policy documents are the single source of truth for all compliance logic. Every rule, every threshold, every MCC range, every COA requirement lives in one place — not scattered across code branches, tier tables, and analyst spreadsheets.

Merchants carry policy_ids[]. The compliance team publishes policies. Engineering builds the renderer/evaluator once. New industries = new policy doc, not new code.

This replaces the never-implemented QP-3603 / QP-3604 approach that tried to hard-code industry logic into the platform. It's simpler, faster, auditable, and — critically — aligns directly with Mastercard MMSP certification requirements.

The one-sentence version: Compliance writes the rules in JSON. Q reads the JSON and evaluates evidence. Every industry, every merchant, every report flows through the same spine.
02 How It Works — simplified flow 7 steps, one loop ▶
① Author
→
② Assign
→
③ Scan
→
④ Evaluate
→
⑤ Report
→
⑥ Evidence Loop
→
⑦ Bulletin

Author: Compliance writes/updates a policy in the editor or wizard.
Assign: Auto via Q-Trust intake, or analyst manual, or parent inheritance.
Scan: Website, product, COA, license, reputation, MRI pipelines produce evidence tagged to checklist items.
Evaluate: ComplianceCoverageValidator + Q AI read policy + evidence → pass/fail/needs-review.
Report: Single button generates a Q-authored PDF. SOC 2 audited.
Evidence Loop: Findings flow into remediation, which triggers re-scan. Closed loop.
Bulletin: Policy version changes notify merchants automatically. Repeat.

06 Mastercard Certification Path MMSP requirements → policy-first mapping ▶

Mastercard's Merchant Monitoring Service Provider (MMSP) certification requires that every monitored merchant's compliance posture be documented, versioned, reproducible, and auditable. Policy-first architecture satisfies each of these by design, not as an afterthought.

▪ MMSP Requires
1.
Documented monitoring criteria — the exact rules applied to each merchant.
2.
Versioned policy history — what was in force when a decision was made.
3.
Reproducible outcomes — same inputs → same outputs, years later.
4.
Evidence retention — raw signals kept for audit window.
5.
MCC-aware categorization — merchant category codes drive rule selection.
6.
Human review gate — humans approve high-risk determinations.
▸ How Policy-First Maps
1.
pl_policy rows are the documented criteria. Each rule = a row, each field = a setting.
2.
Version column on pl_policy. effective_at + immutable history.
3.
pl_compliance_coverage_audit stores policy version × prompt version × model ID per evaluation.
4.
Evidence tables (scans, COAs, licenses) retained; audit row links to exact evidence snapshot.
5.
product_classification + MCC ranges are top-level policy keys.
6.
Approval gate (Decision #3) + analyst override flows make human-in-the-loop explicit.
Bottom line: MMSP certification pressure is not a tax — it's the business catalyst that justifies building the policy-first spine now.
07 Multi-Industry Scalability New industry = publish, not release ▶

The test of the architecture: can we add a new vertical without an engineering release? With policy-first, the answer is yes.

  1. Publish policy document. Compliance writes a new policy in the editor (e.g. peptides-v1.json).
  2. Assign merchants. Existing peptides merchants get the policy via bulk assignment or intake re-run.
  3. Configure scans. Scan pipelines read the new policy's checklist and mri_configuration — no new code paths.
  4. Generate reports. Q AI reads the new policy and existing evidence; reports render in the standard format.

Peptides is already scoped (75 requirements mapped). Firearms, medical devices, nutraceuticals, gaming — each will follow the same path. Zero engineering per vertical beyond the initial platform build-out.

08 Q AI Integration Model governance · SOC 2 alignment ▶

Q is how we turn a policy document + evidence into a verdict. Every evaluation Q performs produces a reproducible artifact.

▸ Q integration pattern

Input: policy JSON vN + evidence snapshot + prompt template vM. Output: structured JSON with per-rule pass/fail/needs-review + citations pointing back to evidence. Everything logged.

Model selection
Pinned model ID, never "latest". Example: claude-opus-4-7. Upgrades go through a governance step, not automatically.
Web search
Enabled for regulatory lookups (FDA warning letters, state licensing). Every search query logged.
Prompt versioning
Prompt templates are versioned separately from policies. Audit row stores both IDs so we can replay exactly.
SOC 2 audit trail
Raw Q input/output retained in pl_compliance_coverage_audit. Retention window per Decision #9.
Review queue
Any Q verdict marked needs_review routes to analyst. Override is logged with reason.
Cost guardrails
Prompt caching on policy document reduces per-evaluation cost. Batch mode for scheduled re-evaluations.
Why Q and not a rules engine alone? Rules engines handle deterministic checks (COA presence, license expiration date). Q handles the judgment calls (does this disclaimer actually communicate the required warning?). Both run — rules engine first, Q on what's left. The evaluator orchestrates.

buildDisposition prerequisite bug High Phase 0 fix. Today the code always emits a generic VIOLATIVE enum. The POLICY_VIOLATION enum already exists and is unused — switching is mechanical, but untested code paths will surface.
Color / font mismatches between mockups and production Medium Decide early: mockups adopt production (#1E3A8A + Uber Move) or production adopts mockup tokens (#1E3A5F + Plus Jakarta). Either way — not both. Ship a token file in Phase 1.
Multi-policy merging semantics ambiguous Medium Decision #1 tomorrow. Pick one (strictest-wins is the safest default) and document the escape hatch (explicit priority ordering when needed).

Policy authoring friction (compliance can't use the editor) Low Wizard mode (QP-3703). Form mode (QP-3647). JSON mode as escape hatch. Live preview of rendered rules. Onboard with one vertical at a time.
Estimate optimism Low All estimates assume AI-assisted engineering. Build buffer into Phase 3 & Phase 6 (the two largest). Re-plan after Phase 1 ships.
