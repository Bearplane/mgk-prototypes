Start here · Plain English for non-technical readers Today, compliance rules are scattered across code, tier tables, and analyst spreadsheets. Policy-first puts every rule in one document — a reusable "rulebook" our compliance team can edit directly. Our AI reads that rulebook, scans the merchant's website, catalog, lab reports, and licenses, then writes a compliance report automatically. What this unlocks: (1) Mastercard certification path — their certification requires exactly this architecture. (2) New industries in days, not release cycles. (3) Less manual analyst work — the AI drafts the first pass. (4) Merchants get a self-service dashboard.
01 Executive Summary What "policy-first" means ▶
In plain English One digital rulebook per industry, written by our compliance team. Today, the rules for hemp, tobacco, peptides, etc. live in code that only engineers can change. With policy-first, compliance edits a document; the system reads it and applies it to every merchant in that industry. New regulations ship the day the rule changes — not the next release cycle. How it reduces cost: Less engineering time per new vertical. Less analyst spreadsheet work. Faster regulatory response. Clearer audit trail for Mastercard, SOC 2, and state regulators.

Policy-first means: versioned JSON policy documents are the single source of truth for all compliance logic. Every rule, every threshold, every MCC range, every COA requirement lives in one place — not scattered across code branches, tier tables, and analyst spreadsheets.

Merchants carry policy_ids[]. The compliance team publishes policies. Engineering builds the renderer/evaluator once. New industries = new policy doc, not new code.

This replaces the never-implemented QP-3603 / QP-3604 approach that tried to hard-code industry logic into the platform. It's simpler, faster, auditable, and — critically — aligns directly with Mastercard MMSP certification requirements.

The one-sentence version: Compliance writes the rules in JSON. Q reads the JSON and evaluates evidence. Every industry, every merchant, every report flows through the same spine.
02 How It Works — simplified flow 7 steps, one loop ▶
In plain English Write the rules once. Scan the merchant. Let the AI decide. Tell the merchant. Repeat. Seven steps, one continuous loop. Each time the rules change, every merchant is automatically re-checked and notified. The compliance team never waits for an engineering release to put a new rule in force.

Author: Compliance writes or updates a rulebook in the editor or wizard.
Assign: New merchants are matched to rulebooks automatically via Q-Trust intake; analysts can override; franchises inherit from parent org.
Scan: Robots check the website, product catalog, lab test PDFs, state licenses, reputation signals, and sanctions watchlists.
Evaluate: The system confirms every required check ran, then Q AI reads the rulebook + evidence and decides pass/fail/needs-review.
Report: One button produces a full compliance report PDF — AI-drafted, SOC 2 audited.
Evidence Loop: Findings flow into merchant remediation, which triggers a re-scan. Closed loop.
Bulletin: When rules change, merchants are automatically notified. Cycle repeats.

06 Mastercard Certification Path MMSP requirements → policy-first mapping ▶
In plain English Mastercard certification isn't extra work — it's the same work. Their MMSP certification demands exactly the capabilities we're building: documented rules, versioned history, reproducible AI decisions, and a complete audit trail. Finishing the policy-first spine IS the certification path.

Mastercard's Merchant Monitoring Service Provider (MMSP) certification requires that every monitored merchant's compliance posture be documented, versioned, reproducible, and auditable. Policy-first architecture satisfies each of these by design, not as an afterthought.

▪ MMSP Requires
1.
Documented monitoring criteria — the exact rules applied to each merchant.
2.
Versioned policy history — what was in force when a decision was made.
3.
Reproducible outcomes — same inputs → same outputs, years later.
4.
Evidence retention — raw signals kept for audit window.
5.
MCC-aware categorization — merchant category codes drive rule selection.
6.
Human review gate — humans approve high-risk determinations.
▸ How Policy-First Maps
1.
pl_policy rows are the documented criteria. Each rule = a row, each field = a setting.
2.
Version column on pl_policy. effective_at + immutable history.
3.
pl_compliance_coverage_audit stores policy version × prompt version × model ID per evaluation.
4.
Evidence tables (scans, COAs, licenses) retained; audit row links to exact evidence snapshot.
5.
product_classification + MCC ranges are top-level policy keys.
6.
Approval gate (Decision #3) + analyst override flows make human-in-the-loop explicit.
Bottom line: MMSP certification pressure is not a tax — it's the business catalyst that justifies building the policy-first spine now.
07 Multi-Industry Scalability New industry = publish, not release ▶
In plain English New industry = new rulebook, not new product release. Today, launching peptides or firearms would be a multi-month engineering project. With policy-first, compliance writes the rulebook, assigns it to merchants, and the existing scanners and reports just work. Our time-to-launch for a new vertical drops from quarters to days.

The test of the architecture: can we add a new vertical without an engineering release? With policy-first, the answer is yes.

  1. Publish policy document. Compliance writes a new policy in the editor (e.g. peptides-v1.json).
  2. Assign merchants. Existing peptides merchants get the policy via bulk assignment or intake re-run.
  3. Configure scans. Scan pipelines read the new policy's checklist and mri_configuration — no new code paths.
  4. Generate reports. Q AI reads the new policy and existing evidence; reports render in the standard format.

Peptides is already scoped (75 requirements mapped). Firearms, medical devices, nutraceuticals, gaming — each will follow the same path. Zero engineering per vertical beyond the initial platform build-out.

08 Q AI Integration Model governance · SOC 2 alignment ▶
In plain English Q is our AI evaluator. It reads the rulebook, reads the evidence we've collected about a merchant, and writes a reasoned verdict — with receipts. We pin the exact AI model version (never "latest") and log every input and output, so an auditor three years from now can replay any decision we made. Splitting work: the rules engine handles black-and-white checks (Is the license expired? Is the COA present?). Q handles judgment calls (Does this disclaimer actually convey the required warning?). Both run; the evaluator orchestrates them.

Q is how we turn a policy document + evidence into a verdict. Every evaluation Q performs produces a reproducible artifact.

▸ Q integration pattern

Input: policy JSON vN + evidence snapshot + prompt template vM. Output: structured JSON with per-rule pass/fail/needs-review + citations pointing back to evidence. Everything logged.

Model selection
Pinned model ID, never "latest". Example: claude-opus-4-7. Upgrades go through a governance step, not automatically.
Web search
Enabled for regulatory lookups (FDA warning letters, state licensing). Every search query logged.
Prompt versioning
Prompt templates are versioned separately from policies. Audit row stores both IDs so we can replay exactly.
SOC 2 audit trail
Raw Q input/output retained in pl_compliance_coverage_audit. Retention window per Decision #9.
Review queue
Any Q verdict marked needs_review routes to analyst. Override is logged with reason.
Cost guardrails · ~85k tokens/report
Each compliance report is ~85k tokens of Q input/output. At thousands of merchants × monthly cadence this compounds — mitigate with prompt caching on the policy document (~90% hit rate), batch mode for scheduled re-evaluations, and per-evaluation cost telemetry with budget alerts.
Why Q and not a rules engine alone? Rules engines handle deterministic checks (COA presence, license expiration date). Q handles the judgment calls (does this disclaimer actually communicate the required warning?). Both run — rules engine first, Q on what's left. The evaluator orchestrates.

10 Risks & Mitigations What could derail the plan ▶
In plain English Seven risks, ranked. The two high-risk items both block Phase 3 (the flagship AI reports). Everything else is manageable with clear decisions tomorrow.
RiskLevelMitigation
CIO's SOC 2 concerns unresolved High Decision #9 tomorrow. Lock retention window, replay mechanism, access controls before Phase 3 starts. Q integration work blocks on this — don't start without clarity.
No way to test a policy before publish High Today there's no sandbox to preview how a policy change will score real merchants before it goes live. CIO raised this as a SOC 2 concern — an incorrect publish would cascade to every affected merchant. Ship a policy sandbox alongside Phase 3: evaluate a candidate policy against N sampled merchants, produce a diff vs. current, require sign-off before publish.
No merchant-facing dashboard yet (client.tsx is a stub) High The merchant portal bundle exists as a shell — no compliance surfaces inside it. Phase 7 delivers the real dashboard. Decisions #5 and #6 shape what merchants will see and do. Until then, every merchant question routes to an analyst.
buildDisposition prerequisite bug High Phase 0 fix. Today the code always emits a generic VIOLATIVE enum. The POLICY_VIOLATION enum already exists and is unused — switching is mechanical, but untested code paths will surface.
Color / font mismatches between mockups and production Medium Decide early: mockups adopt production (#1E3A8A + Uber Move) or production adopts mockup tokens (#1E3A5F + Plus Jakarta). Either way — not both. Ship a token file in Phase 1.
Multi-policy merging semantics ambiguous Medium Decision #1 tomorrow. Pick one (strictest-wins is the safest default) and document the escape hatch (explicit priority ordering when needed).
Q model version drift Medium Pin explicitly. Never call "latest". Governance gate on model upgrade. Regression test set of known policies + evidence pairs with expected verdicts.
Operational complexity: 7 Bull queues + 23 cron jobs Medium Compliance work is already fanned out across 7 Bull queues and 23 scheduled cron jobs. Policy-driven scans/evaluations add surface area. Phase 5 introduces a unified scheduler abstraction; don't add a new queue without retiring one. Monitor queue depth and job-success rate centrally.
No monitoring control center for automated scans Medium Today there's no single pane-of-glass showing which merchants were scanned, when, and what failed. Monthly scan cadence will break quietly without this. Phase 7 includes an ops dashboard: scan coverage per merchant, queue health, exception routing. Ship it before scaling merchant count.
AI evaluation cost at scale Medium A full report evaluation is ~85k tokens of Q input/output. At thousands of merchants × monthly cadence, cost compounds quickly. Mitigations: prompt caching on the policy document (~90% hit rate expected), batch mode for scheduled re-evaluations, per-evaluation cost telemetry + budget alerts.
Policy authoring friction (compliance can't use the editor) Low Wizard mode (QP-3703). Form mode (QP-3647). JSON mode as escape hatch. Live preview of rendered rules. Onboard with one vertical at a time.
Estimate optimism Low All estimates assume AI-assisted engineering. Build buffer into Phase 3 & Phase 6 (the two largest). Re-plan after Phase 1 ships.

12 Architectural Strengths Why this plan lands — 10 reasons ▶
In plain English We aren't starting from scratch — most of the pieces are already in place. The hooks exist, the rule processor is installed, the enum vocabulary is defined. Policy-first threads them together into a coherent spine that the compliance team can drive without engineering touching code.
01

json-rules-engine is already in the codebase

The open-source rule processor is already listed in package.json — we just haven't wired it up. Phase 1 flips the switch. No new vendor, no new dependency, no procurement cycle.

02

resolveCoaValidationRule() is a single-line swap

One function decides which COA tests to require. Today it reads from the tier config table; switching it to read from the policy document is literally a one-line change. It's the first concrete proof that policy-first works.

03

Net-new infrastructure is small

The whole architecture needs just 4 new tables, 2 new columns, and 3 new services. Everything else reuses existing hooks. Small surface area means fast to build, fast to review, fast to test.

04

Two-layer storage is SOC 2 friendly

Every policy is stored as both a human-readable narrative and a machine-readable JSON. Auditors get the prose; systems get the data. No sync burden — the narrative and JSON are the same record, different views.

05

Q AI evaluation loop is novel and scalable

Combining a deterministic rules engine (for black-and-white checks) with Q (for judgment calls) on the same policy document is a pattern we haven't seen in market compliance tooling. It scales: add a rule, both layers pick it up.

06

New industry = new policy doc, not new code

Peptides, firearms, nutraceuticals — each is a published JSON document, not a platform release. Time-to-launch for a new vertical drops from quarters to days. The moat grows with every policy published.

07

Existing hook points are clean

The POLICY_VIOLATION enum is already defined (just unused). The pl_compliance_element table already models findings. Phase 0 wires these up — we're not inventing vocabulary, we're activating it.

08

Version bumps are policy edits — zero engineering

Move a threshold from 95% to 97%? Edit the policy, publish v2, done. No pull request, no QA cycle, no release. Compliance ships changes the same day regulations change.

09

Bulletins give merchants a grace period

When a policy updates, merchants get a bulletin with an explicit grace period before the rule is enforced. Nobody gets blindsided. Customer-friendly and defensible in a dispute.

10

Compliance iterates without code deploys

The entire rule lifecycle — author, review, publish, retire — runs in the policy editor. Engineering's job ends at "the editor works"; compliance owns what it says. That's the decoupling the business has been asking for.

Bottom line: policy-first isn't a rewrite. It's a small, high-leverage set of activations that connects the parts we already have into a spine the business can drive directly.
