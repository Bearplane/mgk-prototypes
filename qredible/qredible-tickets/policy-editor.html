Checklist items are generic policy constraints — licenses, attestations, and document-level rules — that become required/conditional items at onboarding and renewal. COA requirements are handled separately because they're enforced at the Product Classification level (see below), not as standalone checklist items.

🗄 Real data model — this isn't a new abstraction. COA requirements already live on the existing pl_category table (Product Classification) as the coa_requirement enum column: REQUIRED / CONDITIONAL / NOT_REQUIRED. Each product's effective requirement is resolved via pl_product_category assignments, sorted by priority, in ProductHelper.determineCOARequirement(). The policy editor doesn't create a parallel concept — it sets the enum value on matching classification rows when the policy publishes. Admins can already edit this field directly in the Product Classification admin tile; policies are the programmatic way to set it in bulk across a merchant cohort.
Why this is one edit instead of three tickets — Pre-pivot, adding a new compliance rule meant (a) a new attribute row in the Regulatory Attributes table, (b) a rule definition in the Compliance Rules Engine, and (c) a code change to map it into the scanner. Now it's one line in the policy doc, versioned by the editor, and every downstream system (scanner, onboarding, MRI, reports) reads it from the same source. For COA requirements specifically, the policy writes directly into the existing pl_category.coa_requirement field that the rest of the platform already consumes.

⑤ COA Requirement — How It Actually Works
A policy's COA rule drives PlCategoryEntity.coaRequirement
COA requirement does not live on individual products — it lives on Product Classification rows (PlCategoryEntity.coaRequirement). The enum values are: REQUIRED / CONDITIONAL / NOT_REQUIRED (CategoryCoaRequirement enum in the API). Products inherit from their classification at runtime via ProductHelper.determineCOARequirement(), which iterates categories by priority.

A policy's COA rule is a governance layer that drives these classification rows. The Policy Editor sets the rule; the Product Classification admin tile (Q-Audit Global Settings) remains the operational source of truth at runtime. The runtime does not change — it keeps querying PlCategoryEntity.coaRequirement exactly as it does today. Priority awareness matters: an override story must account for the priority iteration order when two classifications conflict.
REQUIRED
COA must be on file and current for all products in this classification
CONDITIONAL
COA required when merchant.type or other policy condition is met
NOT_REQUIRED
Product classification does not require a COA for this category
