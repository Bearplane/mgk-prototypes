Policy Evidence Loop — Closed-Loop Compliance Detection

The system that connects scan findings back to policy assignment — detecting missing policies based on concrete evidence the platform obtains about a merchant. Worked example: Acme Hemp Co caught selling kratom while only on hemp-CBD policies.

① The Big Idea
🔍 Policies define what we look for. Findings tell us what we found.

Policies define what we look for. Findings tell us what we found. If we find evidence of a regulated activity that none of the merchant's assigned policies cover, that's a coverage gap — and the system should catch it automatically and suggest (or auto-attach) the missing policy. The merchant's policy stack should reflect what they actually do, not just what they said at sign-up. This closes the loop: every scan not only evaluates the merchant against their current policies, it also checks whether their policy stack is still complete based on what the evidence shows.

Each finding gets stamped with one or more evidence tags — short, normalized facts about what was observed. Tags are the currency of the coverage check.

Evidence tags are the currency of the coverage check. Tags are not a separate admin surface — they're a fixed, extensible vocabulary of normalized facts. Each tag is consumed by policies in two ways: (a) by the policies a merchant already has, to evaluate compliance, and (b) by the coverage check, to detect missing policies. The tag vocabulary is seeded from existing plComplianceElementEntity.elementName values — no new admin surface needed to bootstrap it.
④ Coverage Check
🔎 Detecting missing policies — the algorithm
How the coverage check works
1
For each evidence tag observed on the merchant in the latest scan batch
2
Find all canonical policies whose expected_evidence_signature includes that tag
3
Compare the matching policies against the merchant's current policy_ids
4
If a matching policy exists in the canonical library but is not assigned to the merchant → coverage gap detected
5
Open a Suggested Policy item in the Q-Audit review queue with supporting evidence and confidence score
⚠ Prerequisite fix required — disposition granularity bug.
CoaHelper.buildDisposition() at src/coa/coa.helper.ts:475 always assigns the generic Disposition.VIOLATIVE regardless of which rule fired. The granular values VIOLATIVE_DELTA9_THC, VIOLATIVE_THCO, and VIOLATIVE_EXPIRED_COA are defined in the enum but never written. The policy engine needs to distinguish which violation caused a VIOLATIVE disposition — this is a prerequisite fix before evidence tagging on COA findings works correctly.
Example — policy signature declaration

Each canonical policy declares the evidence tags that signal a merchant should be subject to it:

// kratom-watchlist-v1.json
"expected_evidence_signature": {
  "match_any": [
    "contains-kratom-keyword",
    "product-category-kratom",
    "analyte-mitragynine-detected"
  ]
}
// Any one of these tags on a merchant
// = strong signal they should be on kratom-watchlist-v1
The key insight: The check runs automatically after every scan. A merchant can't slip through by simply not disclosing a product category — the evidence finds it for us.
Real integration points — where policy-first hooks in
Evidence tagging injection
All Compliance Element findings are created in ScanService.saveCustomComplianceResults() at src/scan/scan.service.ts:~677. One EvidenceTaggingService.tag() call inserted before each PlComplianceFindingEntity save stamps normalized tags on every finding. Same injection point exists in saveComplianceFindingResults() for system findings (FDA, age verification, etc.).

The Compliance Element elementName value is the natural tag vocabulary seed — no new admin surface needed to bootstrap the vocabulary.
Coverage check trigger
ScanService.scanUrl() already emits end-of-scan events via emitter.emit() at ~line 611 (e.g., EventConstant.START_TRANSCRIPTION_JOB_STATUS_CHECKING_CRON). The same pattern fires a new COVERAGE_CHECK event that starts a new ComplianceCoverageCheck.cron.ts after every scan. No new queue or infrastructure — just another cron file in src/scheduler/cron/.

COA validation rule lookup already at resolveCoaValidationRule() in CoaExtractionExportService:~380 — single-line swap to PolicyLookupService call.
Rules engine — dormant dependency
json-rules-engine is declared in RTQ-Admin-API/package.json dependencies but has zero wiring in any service. Someone imported a full rules-engine library and never hooked it up. This is the cleanest path for the coverage check evaluator — wire policy doc rules to json-rules-engine without adding any new dependency.

⑥ Confidence Levels ❓
When to auto-attach vs. suggest vs. log only
Proposed confidence tiers
Tier Criteria Proposed action Example
HIGH 3+ distinct evidence source types (e.g. website scan + product + COA) Auto-attach policy + notify merchant + open bulletin Kratom keyword on site + 15 kratom SKUs + mitragynine on COA
MEDIUM 2 source types, or 1 strong source (COA analyte confirmation) Surface in Q-Audit review queue — analyst must approve before attach Kratom keyword on site + 15 kratom SKUs (no COA yet)
LOW 1 weak source (single keyword hit on one page, no product evidence) Log only — no surface, no queue item. Accumulates toward MEDIUM. One mention of "kratom" in a blog post with no products found
❓ TBD — Auto-attach thresholds: Brian and Ronaldo decision. Auto-attach is a strong action: it changes the merchant's compliance requirements without their explicit confirmation. Analysts may want veto authority on every policy change, even HIGH-confidence ones. Alternatively, analysts could be notified after auto-attach with a 24-hour veto window. These thresholds should be tuned against real data once the first scan cohort runs through the coverage check.
⑦ The Reverse Case 🔁
🔁 Detecting policies that no longer apply
The scenario

A merchant currently on tobacco-vape-v1 removes all vape products from their catalog. Their product scans no longer show any vape-related evidence. Should the system detect that the policy may no longer apply and suggest removing it?

Recommendation: suggest-only, never auto-detach. Removing compliance coverage is inherently riskier than adding it. Proposed behavior: if zero matching evidence appears across 90 consecutive days of scans (not just one scan cycle), surface a "Policy may no longer apply" suggestion in Q-Audit. Analyst reviews and explicitly detaches. Never auto-detach.
Asymmetry is intentional. Attaching a missing policy = reducing compliance risk. Detaching a policy = increasing compliance risk. The system should be more cautious about detach suggestions than attach suggestions, and should always require human sign-off on removal.

⑨ Open Questions ⚠️
What's not decided yet — specific to the evidence loop
1. Auto-attach vs. suggest-only thresholds — Brian / Ronaldo decision
The proposed tiers (HIGH = auto-attach, MEDIUM = suggest, LOW = log) are a starting point. These thresholds should be calibrated against real data and analyst feedback. The risk tolerance question: is a wrong auto-attach (attaching a policy to a merchant who doesn't need it) more costly than a wrong miss (failing to catch a coverage gap)?
2. Evidence tag vocabulary governance — who owns the canonical tag list, how does it evolve?
New scan capabilities will produce new evidence types. Who decides when a new tag is added to the vocabulary? Who maps new tags to existing policy signatures, and who decides when a new tag warrants a new policy? This needs an owner and a lightweight change process.
3. False positive handling — how do merchants challenge a wrongly-suggested policy?
A merchant selling a product with "kratom" in the name as a colloquial ingredient reference (not actual kratom) may be flagged incorrectly. How do they challenge the suggestion? What's the analyst review process? What's the escalation path if the merchant disputes the evidence that triggered the suggestion?
4. Reverse case (auto-detach) — keep or punt for v1?
The recommend is suggest-only for policy removal, never auto-detach. But even suggest-only requires a 90-day evidence window and analyst review workflow. This may be a v2+ feature — the attach direction is more urgent and more valuable. Confirm scope for v1.
5. Cross-merchant signal — can many merchants' evidence suggest a new policy that doesn't exist yet?
If a significant number of merchants consistently produce evidence of a regulated activity with no matching canonical policy, that's a signal that a new policy should be created. This "long-tail discovery" capability is a v2+ idea — but worth designing the evidence data model to support it from the start.
6. SOC 2 audit trail — every coverage decision must be auditable
Which evidence tags triggered the suggestion, which scan run produced them, who reviewed the suggestion, what action was taken, and when — all of this needs to be captured in an immutable audit log. This affects the data model for coverage suggestions and must be designed before the feature ships.
7. Coverage check frequency — after every scan, or batched nightly?
Running the coverage check after every individual scan is the most responsive option but may be noisy (a single partial scan with one keyword hit doesn't justify a queue item). Batching per scan session (all scan types for a merchant, collected over a rolling 24h window) may produce better signal-to-noise. Per-merchant vs. platform-wide batch cadence is also a performance question.
