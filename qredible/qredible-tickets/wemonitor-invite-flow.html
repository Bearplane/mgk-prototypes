We Monitor · Invites tab — unified "Send Invite" flow

QP-3731 Invites tab adds a stacked Activity Dates column (Sent / Joined / Last Active) and drops License Status. One "Send Invite" button handles both single and multi-row invites — no separate Bulk Invite modal. At invite-send time a single "MyCOA invite" toggle decides whether the recipient lands in the MyCOA setup wizard (ON) or the standard post-join flow (OFF).

ℹ DESIGN NOTES — NOT IN BUILD
  • Columns: checkbox · Legal Name · Email Invite Status · Company Status · Activity Dates · Team Members · Actions. License Status is removed. Inner Active/Pending sub-tabs are removed (Company Status pill carries that state inline).
  • No per-row Feature column. Scope is intentionally simple for v1 — a single MyCOA toggle lives on the invite-creation surface, not on every row.
  • MyCOA invite toggle (on the Invite dialog): ON → recipient is routed into the MyCOA setup wizard after joining, before the paywall. OFF → standard post-join flow, no wizard. One boolean, stamped on the invite token at send time.
  • Activity Dates is a 3-line stack: Sent: MM/DD/YYYY · Joined: MM/DD/YYYY · Last Active: MM/DD/YYYY. Missing values render as —.
  • Email template modal is a single modal with Subject, Body, CTA, and standard merge tags ({{sender_name}}, {{recipient_first_name}}, {{company_name}}). No per-feature token — the template is MyCOA-agnostic for v1.
  • Single "Send Invite" entry point — no separate Bulk Invite modal. The toolbar always shows one Send Invite button. When no rows are selected it opens the single-merchant modal ("Invite merchant"). When N rows are selected in the table, the button label adapts to Send Invite (N) and opens the batch modal listing those selected rows — no CSV upload, no manual-entry tab. One code path, two modes. The same MyCOA invite toggle concept applies at batch level.
  • Invite scoping: Free-form Legal Name + Email entry is intentionally excluded. The single-invite and bulk-invite paths are both scoped to existing WeMonitor companies in this iteration. Free-form invites for brand-new companies are deferred. To invite a company not yet in WeMonitor, add it first.

Bulk action bar (Resend / Revoke) vs Send Invite button:
  • Send Invite (N) in the toolbar — opens the "Invite N merchants" modal for the selected rows. This is the only invite entry point, single or bulk.
  • Resend Invites — enabled when every selected row has status Sent or Expired. Regenerates the token and re-emails using the current template.
  • Revoke Invites — enabled when every selected row has status Sent or Expired. Invalidates the token; accepted invites are never revoked from this surface.
  • If selection is mixed (e.g. includes a Joined row), incompatible bulk actions disable with a tooltip explaining why.

Key differences between modes:
  • Single invite — company picker (typeahead). The operator must select a company already in WeMonitor; free-form Legal Name + Email entry is not available.
  • Multi-row invite — no free-form entry, no CSV upload. Recipients are pulled directly from the rows already checked in the table. The modal lists them with a per-row × to remove any before sending.
  • Both modes use the same email template and the same MyCOA toggle concept. The only structural difference is where the recipient list originates.
  • Duplicate detection runs against existing active invites for the same email address. Duplicates are shown inline and skipped at send time — no blocking pre-send confirmation dialog needed.

QP-3731 · Invite scoping
How invite scoping works

Both invite paths are scoped to existing WeMonitor companies.

The single-invite dialog uses a company picker (typeahead) — the operator must select a company already in WeMonitor before they can send. The bulk-invite dialog (multi-row from table selection) populates from the WeMonitor table selection — same constraint applies.

All invite paths are scoped to existing WeMonitor companies in this iteration. Free-form invites for brand-new companies are deferred.

QP-3731 · Recipient flow (MyCOA toggle ON)
What the invited merchant sees when MyCOA is ON
When MyCOA is toggled ON at invite-time:
  1. The recipient accepts the invite and completes account signup.
  2. They are routed directly into the MyCOA setup wizard — upload first COA, display preferences, domain/CNAME intent.
  3. On wizard completion they hit the paywall, filtered to MyCOA-eligible plans (plan→feature mapping lives in Q-Config, QP-3732).
  4. Plan selection finalizes the MyCOA subscription; the published MyCOA page goes live.
