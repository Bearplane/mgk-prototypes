QP-3465 / QP-3526 — Inactive COA Detection & Audit View
Introduces runtime "inactive" flagging for expired COAs that have valid replacements, and an Audit View that filters them out of compliance calculations.
Section 0 — The Problem

Why This Exists

Today, any expired COA on a product can cause it to show Violative — even if a perfectly valid, current COA also exists on that same product. This creates false positive violations that erode trust in compliance data and generate unnecessary manual review work.

  • Product A → has 1 expired COA and 1 valid COA → today shows Violative ← this is the problem
  • Product B → has only expired COAs → correctly shows Violative
⚠️ Product A — False Positive (Current Behavior)
Batch 2026-A — valid.pdf ✓ COA Validated
Batch 2025-A — expired.pdf ⚠ Expired
Today's system result
🔴 Shows: Violative — expired COA poisons the product status
With this fix (Audit View)
🟢 Shows: COA Validated — expired COA is inactive (has valid replacement)
🚨 This is the false positive. The product is genuinely compliant — a valid COA exists — but the system calls it Violative because of a superseded expired batch.
✅ Product B — Correctly Violative (No Change)
Batch 2024-A — expired.pdf ⚠ Expired
Batch 2023-B — expired.pdf ⚠ Expired
Today's system result
🔴 Shows: Violative — all COAs expired, no valid replacement exists
With this fix (Audit View)
🔴 Shows: Violative — no change, still correctly violative
✅ This is correct behavior. No valid COA exists, so Violative is the right call. The fix does not affect this product.
Problem Summary: The current compliance engine doesn't distinguish between "expired and has a valid replacement" vs "expired with no replacement." Both cases cause Violative today. QP-3465 introduces the inactive concept to separate them at runtime, without permanently altering any COA's stored status.
Section 1 — Inactive COA Detection Logic
🔍 Inactive COA — Detection Conditions
1
COA is expired — the expiration date has passed as of today's date.
2
A valid replacement exists — the same product has at least one other COA that is not expired and not violative.
BOTH conditions must be true
✓ Both true → COA flagged Inactive (runtime only, no DB change)
Only condition 1 → COA stays Active (no valid replacement, still relevant)
⚠️ Critical: Expired ≠ Inactive without a replacement

If all COAs on a product are expired and no valid replacement exists, none are considered Inactive. The product remains Violative and those expired COAs must not be hidden — they require attention.

Inactive status only applies when a valid, non-expired replacement COA exists on the same product.

⚡ Runtime flag only — not stored, not archived, reversible
📏 Governing COA Rule
⚖️ Which COA determines compliance status?

When a product has multiple valid (non-expired, non-violative) COAs, the compliance engine uses the COA with the longest remaining validity period as the governing COA for status reporting.

This ensures the most current and longest-valid certification is used, rather than an arbitrary or oldest-first selection.

Example — Product A
Batch 2026-A — valid.pdf Governs ← 12 mo remaining
Batch 2025-A — expired.pdf Inactive (expired)
Note: If two non-expired COAs exist (e.g., Batch 2026 with 8 months remaining, Batch 2027 with 14 months remaining), Batch 2027 governs. The longer-validity COA wins.

📐 Implementation Notes
🔵 Runtime Only
Inactive flagging is computed at query time. No changes to stored COA status fields, no archived state, no permanent records modified.
🔵 View = User Preference
Default / Audit view selection is a UI preference, not a saved filter. Could be persisted per user session or per page state. Orthogonal to existing filter chips.
🔵 Export Behavior
In Audit View, exports reflect only visible (active) data. Supports clean compliance reporting without manual filtering before export.
