Current State — Three-Place Split

Compliance logic is scattered across RTQ-Admin-API TypeScript, Middleware regex/scraping, and GPT-4 prompt fragments embedded in code. No shared registry, no versioning, no company-attribute awareness. Changing a rule touches 4+ files.

What works today (~25 checks)
Already partly policy-driven: COA cannabinoid thresholds (pl_config_param.coa_validation_rule), cannabis-product keyword detection (pl_config_param.product_scan), violation phrase list, Compliance Elements (pl_compliance_element CRUD). These map cleanly 1:1 to individual policies.
What's painful (~12 fragmented checks)
FDA disclaimer detection: 3-way split across SCAN-005 toggle + MID-001 regex bank + MID-101 GPT-4 prompt. Same for age-gate, privacy links, contact info. Adding a new detection method means touching RTQ-Admin-API + Middleware + the GPT-4 prompt — three repos, three deploys, three regression risks.

Where We're Going

Target State — 3-Layer Policy Architecture

Every compliance rule lives in a versioned registry. Company attributes propagate into the resolver before scans run, before UI renders, before submissions are accepted. Compliance team edits rules without engineering deploys.

Target architecture — 3 layers + company-attribute resolver
COMPANY-ATTRIBUTE INPUT (evaluated by resolver before anything runs)
🏢
pl_company — company-level attributes
industry (peptide / cannabis / lending / …) · track (A / B / N/A) · sales_channels (card_present, card_not_present, marketplace, wholesale) · mycoa_subscribed (boolean, derived from subscription table) · extensions JSON (future industry-specific)
↓ feeds into resolver — runs BEFORE scans, BEFORE UI render, BEFORE submit validation ↓

LAYER 2 — Policies (compliance-team-readable; authored in Policy Editor)
2
pl_qsa_policy
Selects checks from registry · per-policy severity overrides · conditional rules: "if company.track == 'B' then check applies with severity=Required"
≈
pl_qsa_policy_version
Versioned parameters JSON + output_schema JSONSchema7 + lifecycle state (DRAFT → TEST → ACTIVE → DEPRECATED). Four-eyes gate at publish.
⬡
pl_qsa_policy_group
Bundles N policies into a vertical: CBD Standard v3, Peptide Track B v1, Online Lending v1. One group → assigned to companies via pl_qsa_assignment.
↓ compiler lowers policies + company attributes into engine-readable form ↓
LAYER 3 — Engine-readable (compiled; never round-tripped)
3
pl_qsa_policy.parameters (compiled)
Generated at policy publish time. Engine reads only this. Source of truth remains Layers 1+2. Includes resolved predicate trees evaluated against company attributes.
▶
PolicyResolverService
Takes (company_id, policy_id) → returns the attribute-evaluated policy: which checks apply, what severity, which workflow branches activate. UI, scanner, and validator all consume this same resolved form.
📋
pl_qsa_compliance_finding
Persistent finding memory with state machine: NEW → ONGOING ↔ RESOLVED → ANALYST_CLEARED. Immutable first_detected_at. Hash-chained audit log for SOC 2.
Resolver runs first
Resolver evaluates company attributes BEFORE scans run. A Track A peptide merchant doesn't trigger COA extraction. A CP-only merchant skips all website checks. No wasted scan compute.
Branching is policy-driven
"COA URL required: true/false" comes from the resolved policy, not a hardcoded switch statement. New workflow branches require only policy changes, not code deploys.
Industry ≠ schema migration
New vertical = define checks + extension fields + compose into policies. The registry's extensions JSON column absorbs industry-specific fields without altering the core table schema.

Decision 4a — Company Attributes Drive Policy Behavior

Why This Matters: Company-Attribute Behavior Matrix

Same products, same checks — completely different operational behavior based on company attributes. This matrix is the load-bearing logic the resolver evaluates before anything else runs.

"I do like the recommendation but want you to have that additional context… fields are so crucial to have be able to add to the company level and inherited by something like a policy because it could completely change the way that the system operates. With Track A versus Track B especially, with the same products requiring a COA or not requiring a COA or the scan check being necessary or not necessary, or the COA URL being required or not based on them having a website and their sales channels. There are so many different variables that we have to account for and even would change the way that workflows funnel and automate themselves or control the way the UI works that it is CRUCIAL we get this right."

— Joseph Floyd, 2026-05-05 morning meeting (Phase-0 walkthrough)

Why this is hard to undo if we get it wrong
Resolved-policy contract: Becomes the API surface for every downstream consumer (UI, scanner, validator). Changing the shape later means coordinated updates across all of them. Lock the shape early.
Company attributes at onboarding: Attributes not captured at company creation require retroactive backfill. We need to know on day 1 which attributes are policy-relevant so we collect them upfront.
Policy Editor UI: Compliance team must author "this check applies if company.track == 'B'" as easily as they set a severity level. The predicate grammar must be designed for non-engineers.
State transitions: If a company changes attributes (Track A → Track B, or CP-only adds a website), the system must re-resolve the policy and potentially re-trigger scans, re-render UI, re-validate open cases.

2DESIGN

Phase 2 — Resolved-Policy Contract

Define the exact JSON shape that every downstream consumer (UI, scanner, validator) reads after the resolver runs. This is the most important API design decision in the migration — lock the shape early because changing it later requires coordinated updates across all consumers.
PARALLEL WITH PHASE 1
Resolver interface resolvePolicy(company_id, policy_id) → ResolvedPolicy — the single function every consumer calls
ResolvedPolicy shape checks[] (which apply), severities, workflowFlags (coaRequired, urlRequired, scanRequired), uiDirectives
Conditional rule grammar Minimal at first: equality + membership predicates against company attributes. JSON predicate tree extensible without breaking existing policies.
UI directive spec Explicit fields: showCoaColumn, coaUrlEditable, showMyCOAUpsell, hideScanFindings, etc.
⚠️ Dependency: Must be ratified by Joseph + Ronaldo before Phase 3 begins. All downstream consumers implement against this contract.
