MRI Check Marketplace — Merchant Risk Intelligence

Browse, enable, and configure third-party data enrichment checks. Each check produces risk signals against your merchant portfolio. Configure confidence thresholds and auto-dismiss rules to eliminate false-positive noise — without losing genuine risk signals.

① Context
The false-positive problem — and how this UI solves it
The problem at scale
MRI checks run against thousands of merchants. At default sensitivity settings, a check like Vital4 will produce ~340 flags per day — the vast majority of which are fuzzy name matches with no real risk. Without a confidence threshold and structured review workflow, compliance staff are buried in noise and cannot triage real risks.
Without this UI:
• ~340 Vital4 flags/day at default threshold 0.3
• No way to auto-dismiss known-low-quality matches
• No SLA on human review — flags pile up
• No explainability — why did this merchant match?
• Staff disable checks entirely to cope with noise
With this UI (threshold 0.72):
• ~12 flags/day — 96% noise reduction
• Auto-dismiss rules handle 230 low-quality matches
• Remaining flags enter SLA-bound review queue
• Every flag includes "why this matched" breakdown
• Check stays on and catches real risk
Two MRI modes: policy_scored — check result contributes a score to the merchant's policy evaluation (pass/fail). risk_intelligence — check result is surfaced as an informational signal in the merchant risk view, does not directly drive policy outcome. Both modes support confidence thresholds and review queues.

⑤ Open Questions
Items for Brian / Ronaldo alignment
Q1 API call cost budget — who owns it?
Each MRI check has a per-call cost (e.g., Vital4 Enhanced Background is priced per lookup, Visa Chargeback Signal is volume-tiered). At scale across thousands of merchants with daily/weekly refreshes, API costs are significant. Q: Is there a per-policy or per-processor API cost budget? Should the marketplace show estimated monthly cost at current tier configuration? Who approves enabling a new check?
Q2 Vital4 API outage during a scheduled scan
If Vital4's API is unavailable when a scan triggers (e.g., maintenance window, rate limit), what is the fallback behavior? Options: (a) skip Vital4 for this scan and mark as "check skipped — API unavailable", (b) retry up to N times then skip, (c) hold the scan incomplete until Vital4 responds. For policy_scored mode, skipping a required check could leave the policy evaluation undefined — this needs an explicit policy (pun intended).
Q3 Are false-positive dismissals audited, and for how long?
Auto-dismissed matches (via rules) and manually dismissed queue items both remove a Vital4 flag from active consideration. Q: Are these dismissals written to an immutable audit log? What is the retention period (30 days? 7 years for BSA compliance)? Can a dismissed match be re-opened if new evidence emerges? The answer shapes whether this feature can be used in regulated processor workflows.
Q4 MRI result caching strategy and staleness
MRI checks run on a refresh_frequency_hours schedule, but a scan may trigger mid-cycle. Q: Does a scan always use the most-recent cached MRI result, or does it trigger a fresh API call? If cached, what is the maximum acceptable staleness before a scan is considered "stale" for compliance purposes? For OFAC (24h refresh), using a 23-hour-old result on a scan should be fine — but using a 6-day-old result probably is not. This needs a defined staleness policy per check type.
