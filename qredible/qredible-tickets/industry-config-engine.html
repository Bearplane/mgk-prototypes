🧠 Intelligence Architecture
Each evidence source contributes signals. Signals pointing to the same IT/ToC increase confidence. Conflicting signals trigger flags. The system can: auto-classify when confidence is High (multiple sources agree), suggest when confidence is Medium (single source), flag when signals conflict, and propose additions when evidence suggests a company operates in multiple industries. No classification is ever final — it's continuously validated against incoming evidence.
🏷️ Deep Dive

Regulatory Attributes — The Connective Tissue

Regulatory attributes aren't just tags — they're the connective tissue that links otherwise siloed data systems together. An attribute can be attached to anything: a product classification, a COA field, a website finding, an MCC code, a license type, or a company directly. This lets the compliance engine ask cross-system questions that were previously impossible.

Real-World Examples — Attribute-Driven Connections
🚬 FDA ENDS Authorization Check
Classification: Vape/E-Cigarette → attribute: FDA_ENDS_PRODUCT → triggers rule: "IF has FDA_ENDS_PRODUCT AND NOT IN authorization_list → PROHIBITED"
If product NOT on list → PROHIBITED. If on list → COMPLIANT.
🧪 COA Field → License Requirement
COA: Delta-9 THC % > 0.3 → attribute: DELTA_9_THC → triggers rule: "IF DELTA_9_THC > 0.3% → requires STATE_CANNABIS_LICENSE"
Links COA lab data to company licensing — if COA shows THC above hemp threshold, verify cannabis license exists.
💳 MCC Code → Website Compliance Expectations
MCC: 5912 (Drug Store) → attribute: FDA_REGULATED_PRODUCT → triggers scan check: "IF FDA_REGULATED_PRODUCT AND website_missing(fda_disclaimer) → NON_COMPLIANT"
MCC code tells us what the company is — attributes tell us what their website SHOULD have. Missing elements = compliance finding.
🌐 Website Finding → Product Red Flag
Website scan: "Cures cancer" found → attribute: HEALTH_CLAIM_DETECTED → triggers: "IF HEALTH_CLAIM_DETECTED AND claim_type IN [cure, treat, prevent] → flag products for review"
Website violations can cascade to product compliance — if the company is making illegal health claims, their products need review too.
🍄 COA Field + Jurisdiction → Legality Check
COA: Psilocybin > 0 + Company state: TX → attribute: CONTAINS_PSILOCYBIN
Attribute says WHAT it is. Rule says what it MEANS in context. Same attribute, different jurisdictions, different outcomes.

🔗 Why This Matters
Attributes are the query language of the compliance engine. They let you build rules like: "If a company has products with attribute X, in jurisdiction Y, with MCC code Z — then they MUST have license type W and their website MUST show element V." Without attributes as connective tissue, each system (scans, COAs, products, licenses, MCC codes) is an island. With them, the entire compliance dataset becomes queryable and interconnected.
