① Context & Problem
Why this surface exists
What is Track B?

Track B = card-present retail — physical dispensaries and storefront hemp/CBD retailers. Unlike Track A (ecommerce) merchants who have a live website to scan, Track B merchants sell products in-store. Their compliance footprint is their product catalog: the full list of SKUs they carry, with associated product attributes (THC/CBD %, category type, brand, barcode).

The gap today

There is no catalog upload UI in Qredible today. Track B merchants cannot submit their product catalog for compliance review. This is a net-new surface. Once uploaded, the catalog feeds the compliance report generation flow — the same hemp-cbd-base-v3 policy evaluation path used for Track A scans, applied to catalog rows rather than page content.

Scope: This wizard is accessed from the merchant's Catalog tab inside the Qredible merchant portal. The staff-facing admin view shows additional fields (policy slug, review queue assignment) not exposed to the merchant. Both views are represented in this mockup.

③ COA Requirement Per Classification
Classification assignment & COA requirement — post-upload step

After upload, the wizard reads PlCategoryEntity.coaRequirement (backed by pl_category.coa_requirement) for each product classification detected in the catalog. The value determines whether a Certificate of Analysis must be on file before a product passes compliance review.

Schema annotation: PlCategoryEntity.coaRequirement is set at the Product Classification level — not per product, not per merchant. The upload wizard reads this at classification-assignment time and flags which products will require a COA before they pass review. Do NOT derive or infer COA requirement from product attributes; always read pl_category.coa_requirement.

⑥ Open Questions
For Brian alignment
Q1 XLSX column ordering — strict or mapped?
For XLSX uploads, should column order be strictly enforced (columns must appear in template order) or should the column mapping UI always be presented regardless of format? Strict ordering reduces wizard complexity but shifts burden to the merchant to conform to the template. The mockup currently shows the mapping step for all uploads — confirm whether XLSX with a known template header row should auto-skip mapping.
Q2 Multi-location retailers — one catalog per merchant or per location?
Some Track B merchants operate multiple physical locations with different product assortments (e.g., a dispensary group with 4 storefronts, each carrying a different SKU mix). Should the catalog be a single entity per merchant company record, or should Qredible support per-location catalogs? If per-location, the upload wizard needs a location selector and pl_catalog must carry a location_id FK. This has significant downstream impact on compliance reporting (do we report per-location or roll up to company?).
Q3 Removed products — do they trigger automatic compliance status change?
When a SKU is absent from a new upload (i.e., the diff shows it as "Removed"), should its compliance status be automatically set to Inactive / Not reviewed or should it retain its last-known status until explicitly archived? The current mockup shows Removed rows in the diff but does not prescribe a status transition. Brian to confirm: does a removal constitute a "clean pass" action (the product is no longer sold, so compliance risk is gone) or does it need explicit staff acknowledgment?
Q4 Retention policy for prior catalog versions
Each upload creates a new catalog snapshot. How long should prior catalog versions be retained? Options: (a) indefinite with soft-delete, (b) rolling N-version window (e.g., keep last 5 uploads), (c) time-based (e.g., 2 years for regulatory audit trail). The diff feature in Section 4 depends on at least the immediately prior version being available. Confirm retention requirements and whether merchants can self-serve access to prior catalog exports, or if that's a staff-only admin function.
