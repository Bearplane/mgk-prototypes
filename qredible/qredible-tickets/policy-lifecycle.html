Policy Lifecycle — Customer Flow Under Policy-First

End-to-end view: onboarding → policy assignment → merchant dashboard → scan trigger → policy chaining → updates → bulletins → overlay model. Includes the chained-policy stack pattern and explicit open questions for stakeholder alignment.

Why policies exist
⚠ The problem

Processors manage thousands of merchants. The Q-ScanAI scan configuration modal configures one scan at a time — which Compliance Checks, Analyst Review Checks, Compliance Elements, custom keywords, and whitelist keywords apply — but there is no way to save that configuration and reuse it. Every scan is set up from scratch. At scale, this doesn't work.

✅ The policy

A policy is a durable, versioned container for a reusable scan configuration. Create once, assign to a merchant set, and the policy supplies the scan config when scans run. Edit the policy and every future scan across assigned merchants picks up the change. COA governance, evidence loop, and merchant dashboard are all downstream uses of the same container.

① Mental Model
The policy-first architecture in one paragraph

In the policy-first model, a policy is a versioned JSON document authored by Qredible compliance staff. It defines everything the platform needs to know about a category of merchant: which documents to require (checklist), how to classify products (product_classification), what MRI checks to run (mri_configuration), which payment rails are allowed (payment_rails), how to compute scores (scoring_model), what reports to generate (report_configuration), and what enforcement actions to trigger (enforcement_rules). A merchant subscribes to one or more policies — stored as policy_ids on their company record. Every scan, dashboard, report, and enforcement action the platform produces for that merchant is computed directly from their active policies. Merchants never see or configure the JSON; they see only its consequences. Per-merchant exceptions live as narrow overlays on top of canonical policies rather than forks.

🟢 Middle layer — State overlay

⬛ Base layer — Category
hemp-cbd-base-v3

Each layer adds a narrow, specific set of checklist items, scoring rules, and enforcement actions. The effective compliance config is the union of all layers.

Example 2 — Multi-category merchant (4 layers)
🔵 Rail layer
payment-rails-visa-v1
Adds: Visa network rules, prohibited MCCs for research product categories
🟣 Category layer
peptides-research-only-v1
Adds: research peptide catalog check, prohibited human-use claims, required disclaimers
🟠 Watchlist layer
kratom-watchlist-v1
Adds: kratom keyword monitoring, state-banned shipping checks, alkaloid COA reporting
⬛ Base layer
hemp-cbd-base-v3
COA, Farm Bill, classification, scoring baseline

A merchant selling multiple regulated categories stacks one layer per category. Adding kratom to an existing hemp merchant's catalog means attaching one policy, not rebuilding their config.

❌ OLD — One industry row → one giant rules table
Industry type = one enormous shared rule table
Every merchant in the same MCC bucket gets the same config
One ticket to add one check for one category
Rules accumulate as spaghetti — any change is risky
No composability across state, product, or rail dimensions
✅ NEW — Thin stack of named, versioned, narrow policies
Each policy is small, focused, independently versioned
Merchant stacks exactly the layers that apply to them
Add/remove one layer without touching any other
Each layer is readable and auditable in isolation
State overlays, watchlists, and rail configs compose cleanly
❓ TBD — Multi-policy conflict resolution. When two policies in a merchant's stack disagree on the same check (e.g. base says "COA optional for wholesalers", state overlay says "COA required for all"), how does the engine decide? Three options: (A) strictest-wins across all sections, (B) explicit union with per-section override declarations in each policy doc, (C) resolution priority defined at stack-creation time. This is a data model decision that affects the policy JSON schema design.

⑦ Overlay Model
Per-merchant exemptions — proposed architecture
⚠️  PROPOSED — STATUS: TBD    This entire section is a design proposal, not a committed feature. Core question for Brian and Ronaldo: do per-merchant exemptions exist in v1 at all?

The overlay model allows a merchant to receive a narrow exception to a canonical policy requirement without forking the policy itself. The base policy defines the default rule. The overlay — a small JSON document on the merchant record — declares specific exceptions, each with a reason, scope, and granting analyst. The compliance engine evaluates policy + overlay for that merchant only. All other merchants on the same policy are unaffected.

Example — Acme Hemp Co, wholesale-only SKU exemption
LayerRuleScope
hemp-cbd-base-v3 All CBD products must display batch number on label All SKUs
acme-hemp-co-overlays-v1 SKU WHOLESALE-BULK-001 — exempt from batch-label requirement 1 SKU only
Engine result All retail SKUs require batch label · WHOLESALE-BULK-001 exempted Merged
Portal display to merchant: "Exempted for you: 1 product. Reason: wholesale-only SKU not sold to consumers. Granted by: J. Floyd, 2026-04-15."
❓ Open question: Do per-merchant exemptions exist as overlays on top of canonical policies in v1, or do we punt on exemptions entirely and handle them case-by-case in analyst notes? The overlay model adds engine complexity and schema overhead. If exemption requests are rare (estimated < 1% of merchants at launch), it may not be worth the architecture cost until v2.
⑧ Open Questions ⚠️
What's not decided yet — alignment needed from Brian & Ronaldo
1. Overlay model — keep or punt for v1?
Exemptions are needed eventually, but the overlay engine adds schema and logic complexity. If the number of v1 exemption requests is small enough to handle in analyst notes, we can defer overlays until v2. Decision needed before the policy data model is locked.
2. Wizard → policy mapping rule: hand-coded table, configurable, or auto-detected?
Three effort levels with different tradeoffs: hand-coded table (low effort, brittle), Q-Audit config surface (medium effort, flexible), or LLM classification from free-text business description (high effort, high coverage). Each has different maintenance needs and error modes. Needs a decision before wizard design begins.
3. Policy publication approval workflow — who can publish, how is sign-off captured for SOC 2?
Currently the Policy Editor allows any authorized user to publish. For SOC 2 compliance, policy changes that affect enforcement may require a two-person rule or explicit approval record with timestamp. What does the approval workflow look like, and who are the designated approvers?
4. Multi-policy scoring — when policies overlap, how does the score combine?
If a merchant is on three policies with different scoring weights for the same dimension (e.g., COA timeliness), how do the weights merge? Weighted average? Max severity? Most permissive? This is a data model and UX question — merchants need to be able to understand their score and how it was computed.
5. Customer-visible policy explainer — what does the merchant actually see about WHY they're on a policy?
The confirmation screen shows plain-English descriptions at onboarding. But if a policy is auto-attached post-onboarding (e.g., after a scan detects a new product category), what do merchants see? They need a clear connection between what was found and what's now required of them.
6. Self-service misclassification flag — what's the flow to request reassignment?
The "This doesn't look right →" button on the confirmation screen needs a defined path. Does it open a support ticket? Route to a live analyst? Trigger a wizard re-run? What's the SLA for resolving a misclassification dispute, and who owns the outcome?
7. Franchise inheritance — out of scope for v1, but the model shouldn't preclude it
A franchisor merchant may want compliance policies to cascade to child merchant accounts. The current model supports this in principle (a franchise-level overlay or shared policy assignment), but it hasn't been designed. Flag as v2+ consideration when the merchant hierarchy model is built — just ensure the data model doesn't foreclose it.
