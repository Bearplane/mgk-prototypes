MID Management — Merchant Identifier Assignment

WeMonitor surface addition: bulk-assign and manage Merchant IDs per company, scoped to the current processor's relationship.

① Context & Today's State

Today (no UI)

  • MIDs stored on pl_company_dba.merchant_identifier
  • Set exclusively via direct DB writes — no application UI
  • No visibility into which merchants have MIDs assigned
  • Multiple DBAs per company = multiple MIDs possible per company
  • Field is effectively read-only from the application's perspective
  • No audit trail for MID create/update/delete
  • Bulk assignment requires manual DB scripts

After this feature

  • New MID column visible in the WeMonitor merchant table
  • Inline "Manage MIDs" action per merchant row
  • Bulk modal for add / edit / remove per company
  • CSV upload for batch MID assignment at scale
  • All MIDs scoped to the current processor — Processor A's MIDs are never visible to Processor B monitoring the same merchant
  • Scoping enforced via pl_merchant_processor_client.pl_merchant_processor_company_id filter
  • Optional label/note per MID (e.g. "Retail", "Wholesale")

⑤ Processor Scoping — Architecture Note
MID privacy is enforced at the data layer, not the UI layer

How processor isolation works

Each processor's MIDs are siloed by the pl_merchant_processor_client join table — no UI suppression required.
🔗
pl_merchant_processor_client Links a company to a specific processor. Every MID read/write operation filters by pl_merchant_processor_company_id — the FK that identifies which processor owns the relationship.
🔒
Processor A cannot see Processor B's MIDs Processor A can assign MID MC-001 to a company. Processor B monitoring the same company can assign MID VIS-887. Neither processor's MIDs appear in the other's WeMonitor view — no extra row-level security policy needed beyond the existing FK filter.
📋
WeMonitor table query The table only fetches MIDs where pl_merchant_processor_client.pl_merchant_processor_company_id = current_processor_id. The MID column in the UI is therefore always processor-scoped by construction.
🗂️
Multiple MIDs per company (multiple DBAs) pl_company_dba.merchant_identifier is per-DBA. A company with 3 DBAs can have 3 different MIDs under a single processor, all shown as a chip list in the MID column.
⑥ Open Questions
Q1 — CSV format

Final column spec: just company_id + mid_value + label, or should we include dba_id for explicit DBA-level targeting? Without dba_id, the API would need to infer which DBA to write to (e.g. first/primary DBA). Recommend clarifying before implementation.

Q2 — Conflict handling

If a MID value already exists for a given (company × processor) pair, should the upload row be skipped with a warning, or should it overwrite the existing label/note? Current mockup proposes skip-with-warning as the safer default.

Q3 — Audit trail

Are MID create / update / delete events currently audited anywhere? Given the processor-scoping sensitivity and potential SOC 2 obligations, MID mutations should likely be logged to the same audit table used for other compliance data changes. Is that table pl_audit_log or a separate entity?

Q4 — Required vs. optional

Are MIDs required for all merchants under a processor, or optional per processor workflow? Some processors may want to block scan review until a MID is assigned; others may treat it as purely optional metadata. Should this be a per-processor configuration flag?

Q5 — MCC (Merchant Category Code)

No MCC field exists on any entity today. If MCC is eventually needed alongside MID (common in card-brand reporting), should it live in the same "Manage MIDs" modal as an additional field per-MID, or in a separate surface? Flagging early to avoid a schema refactor later.
