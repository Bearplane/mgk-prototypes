Q-ScanAI — Compliance Engine

One configuration-driven engine for every compliance scan, every operational workflow, every regulator-grade audit.

Q-ScanAI replaces hard-coded scanner scripts with a versioned, configuration-driven compliance engine. Compliance teams — not engineers — change rules in a dashboard the same day regulation shifts, while every decision the system makes remains reproducible, isolated from production until approved, cryptographically audited, and persistent across years of scans.

Reproducibility
A scan run today can be replayed tomorrow against the exact same rule versions and produce the exact same result. Nothing is lost or overwritten.

02
For every merchant assigned to it — six things, in order

What Q-ScanAI does

A single engine performs all six steps for every merchant, every scan. Not six tools stitched together — one orchestrated pipeline with one audit trail and one governance process.

1 🔍
Discovers
Walks the website, document folder, license registry, or external API to find every page, file, or record to examine.
2 ✅
Applies
Runs every configured compliance check — FDA disclaimers, SSL certificates, contact info, COA directories, prohibited keywords — against the discovered content.
3 ⚖️
Reaches a verdict
Returns PASS / FAIL / N/A / ERROR for every check, evaluated by a versioned rule engine against the policy output and item context.
4 🧠
Remembers
Updates the merchant's persistent compliance record — first-detected dates, ongoing status, recurrence counts — surviving across every future scan.
5 ⚡
Acts
Fires configured follow-up actions: opens support tickets, emails the compliance team, pushes notifications, updates the merchant record.
6 🔔
Audits
Writes every event to an append-only, hash-chained log. Years later, the entire scan can be replayed and explained to a regulator.

Context Store — 4 scope levels

Typed key-value store shared across all bindings in a scan. API: read / write / list / merge / watch / snapshot. Every entry carries provenance metadata (sourceType, policyVersionId).

item
Scoped to one qsa_scan_run_item. Cleared after the item pipeline completes. Holds extracted DOM elements, AI outputs.
group
Shared across all items in one policy’s group. Running totals for AGGREGATE_ITEMS reducers.
scan
Scan-run-wide. Policy outcomes stored here so downstream policies can branch on prior results.
subject
Persisted to qsa_context_entry; survives across scans. Mutation policy: merge / replace / immutable per key (ADR-046).
Large-Value Handling — threshold: 10 KB

Blobs above 10,240 bytes are never stored inline. The system automatically routes to the correct tier based on size and lifecycle.

≤10 KB — Inline DB
MySQL JSON column
Small context entries, policy parameters, rule outputs. Instantly queryable with JSON_EXTRACT.
>10 KB — S3 Content-Addressed
SHA-256 keyed
Screenshots, HTML bodies, PDF COA files. Prefixes: evidence/ · scan-persistent/ · scan-transient/
Ephemeral — Redis
TTL-bounded
Idempotency keys, action output coordination, cross-worker locks. Never the source of truth.
Reporting Strategy — hybrid approach

Three complementary layers: typed columns for high-cardinality GROUP BY, JSON for handler-specific extras, materialized rollups for dashboard load performance.

Typed columns
severity · is_found · is_compliant · resource_kind · resource_key — indexed, fast for aggregation queries.
JSON payload
output_payload holds handler-specific extras (matched selectors, confidence scores, evidence refs). Not indexed; read for drill-down views.
Materialized rollups
qsa_scan_run_summary · qsa_policy_outcome_rollups · qsa_scan_run_check_type_summary — computed at finalizeScan() for instant dashboard load.
05
Same engine, same audit trail, same governance — three policy_mode values (ADR-033)

Three kinds of work

Q-ScanAI runs compliance checks (PASS / FAIL findings), aggregation rules (child-rollup verdicts), and operational workflows (scheduled side effects). All three share one engine, one audit trail, and one governance process.

policy_mode = DETECTION

Detection Check

Determines whether a subject meets a specific requirement. Verdict recorded in persistent finding memory; first-detected date is immutable.

ProducesPASS / FAIL / N/A findings
Persists inqsa_compliance_finding
Mutating actionsForbidden — gate at promotion
EXAMPLE
"Does every page on the merchant’s website contain a valid FDA disclaimer in the footer?"
policy_mode = AGGREGATION

Aggregation Rule

Rolls up child DETECTION results into a single parent verdict. No handler executes — verdict is derived from child outcomes by a rule (e.g., ALL_CHILDREN_MUST_PASS).

ProducesDerived PASS / FAIL
Persists inqsa_compliance_finding
Mutating actionsForbidden
EXAMPLE
"CBD-CONTACT passes only when Phone, Email, and Address all pass."
policy_mode = OPERATIONAL

Operational Task

Automates a scheduled workflow that has no PASS/FAIL outcome. The durable record is the action invocations — emails sent, fields updated, queues pushed.

ProducesSide effects only
Persists inqsa_action_invocation
Mutating actionsRequired — whitelisted via is_mutating
EXAMPLE
"Every Monday, query Cannaspy for each merchant. If a license is missing, email compliance and update the merchant’s status field."
06
No untested config reaches production — ever

Rule lifecycle & promotion gates

Every policy, rule, action, template, and profile passes through the same controlled promotion process. Every transition is audited; every rejected transition is also audited.

DRAFT
Author & reviewers only
Editable as new versions. Not used in any scan, production or test. Old versions immutable from creation.
submit
→
TESTABLE
Test environment only
Visible in test mode. Test scans write to isolated qsa_test_* tables. Successful runs increment successful_test_count.
activate
→
ACTIVE
Production-visible
Used by all production scans for assigned merchants. Cannot be edited — only superseded by a new version.
supersede
→
DEPRECATED
Read-only history
Replaced by a newer active version. Past scans pinned to this version still resolve correctly. Rollback re-activates it.

🔒 Four promotion gates — all must clear before draft → testable → active

  1. Schema validation (mode-specific)
    DETECTION: check_type contract, item-type compat, execution-pattern coherence, mutating-action ban, rule_kind=PASS_FAIL.
    AGGREGATION: aggregation rule binding present, child policies exist, mutating-action ban.
    OPERATIONAL: at least one MAIN binding present.
  2. Successful test run
    At least one successful test scan completed in the isolated test environment against representative subjects.
  3. Four-eyes approval
    A different person from the author approves the activation. Enforced at API and database CHECK constraint — not just policy.
  4. No circular dependencies
    No cycle exists between this rule’s prerequisites and any other rule it ultimately depends on.
👁
Four-eyes is enforced at the database layer. The chk_four_eyes CHECK constraint refuses any active row where approver_user_id = author_user_id. The application-level guard is the second layer, not the only one.
07
Living compliance memory — not one-shot audit reports

Persistent finding memory

Each compliance issue carries a permanent record. The date it was first seen is stamped once and never changes — not even when the rule that detected it is updated. Status transitions are deterministic and regulator-verifiable.

NEW
First time we’ve seen this finding for (subject, item, policy). Engine sets on first FAIL.
ONGOING
Already seen and still failing. Engine sets on subsequent FAIL.
RESOLVED
Previously failed; the current scan PASSED. Engine sets on PASS after a prior FAIL.
ANALYST_CLEARED
Compliance analyst reviewed and explicitly closed the finding with required notes.
first FAIL → still failing → PASS → analyst clears
Reopen logic. If a RESOLVED finding re-appears in a later scan, it auto-reopens to ONGOING with previous_occurrences++. If an ANALYST_CLEARED finding re-appears and the content hash or policy version has changed, it reopens. If both are unchanged, the analyst's clearance is respected.
SOC 2 INVARIANT

first_detected_at is immutable.

first_detected_at: 2025-08-14T09:13:22Z
Set once, on the NEW transition. Never overwritten — not when a rule is updated, not when a finding is resolved and reopens, not when an analyst clears it. This is the regulatory guarantee that makes "ongoing for 247 days" mean what it says.

Strict isolation at every layer — database, storage, credentials

Multi-tenant architecture

Processor profiles cascade to child merchants. Children can override or exclude inherited policies, but exclusions require a permanent written justification. There are no shared data paths between tenants.

Payment Processor — Parent Tenant

Acme Payments

Profile assigned: CBD Standard v3 · effective Q2 2026
INHERITS
Merchant A — Green Leaf Wellness
Inherits the processor's CBD Standard v3 profile unchanged. All 16 checks apply.
profile_version_id = NULL  ·  floats to active
OVERRIDES
Merchant B — Blue Ridge Hemp Co.
Inherits the profile but pins a custom version of the Payment Methods check (uses ACH-only).
policy_version_id = pinned-v9
EXCLUDES
Merchant C — Cascadia Cash & Carry
Inherits the profile but excludes the Age Verification Gate check — cash-only, no e-commerce surface.
justification_text = required & recorded
🔒 Findings — merchant-scoped Compliance findings are owned by and visible only to the specific merchant. Cross-merchant queries are rejected at the repository layer.
📁 Evidence — merchant-prefix S3 Screenshots, files, and matched-text snippets stored under a merchant-scoped S3 prefix. Inaccessible to other tenants.
🔑 Credentials — vault-scoped API keys and secrets stored in AWS Secrets Manager namespaced per merchant. Cross-tenant credential access is impossible by construction.
✅ Justified exclusions only Excluding an inherited check requires non-empty justification_text — recorded permanently and surfaced in audit reviews.

SOC 2 Type II posture — not bolted on, designed in

Compliance, audit & security posture

Every guarantee in section 1 traces back to a structural choice in this section. Defense in depth, multiple enforcement layers, and no human-removable bypass paths.

📝

Append-only audit log

Every event — configuration change, scan run, action invocation, finding lifecycle — is written once and never modified. Cryptographic hash chaining detects any tampering, including by database administrators.
👁

Four-eyes enforcement

Activating any compliance rule requires a different approver from the author. Enforced at both the application layer (NestJS guard) and the database layer (CHECK constraint) — not just by policy.
🔑

Credential isolation

All secrets stored in AWS Secrets Manager scoped per merchant. No credential is ever stored in the database or in application configuration. Just-in-time injection at action execution time only.
🎉

SOC 2 alignment

The audit log schema, lifecycle controls, and access model are designed to satisfy SOC 2 Type II requirements. Designed in from day one — not retrofitted.
🧪

Test ↔ production separation

Test activity is physically isolated at the database level — separate qsa_test_* tables with no foreign keys to production. Side-effecting actions in test mode are dry-run, sandbox-routed, or blocked entirely.
⚙️

Hourly chain verification

An automated process re-checks the entire audit log hash chain every hour and raises an alert on any discrepancy. A daily cron also re-verifies long-tail entries.
⛔

No DELETE paths

Audit entries, finding records, and version history cannot be deleted by any user or system process. Soft-delete is only available where it preserves the row permanently with a tombstone flag.
🛡️

OWASP-aligned API security

JWT authentication, per-merchant API keys, rate limiting at the API Gateway. All credential references resolved through the Secrets layer — never passed directly through the API.
