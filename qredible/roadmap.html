Policy + Scanner Roadmaps

Two development tracks. Both start the same Monday. Both ship in roughly ten working days. They interlock at a single shared contract — the check code — which turns today's parallel work into a clean merger eight phases from now. This page is the shape of that plan.

Huy's Day 1
Policy Track · opens the week

Schema + resolver — the contract

T1 → T3 · foundation stretch
Why this first? T1 lands the two new tables — pl_policy and pl_merchant_policy — plus TypeORM entities and the TypeScript interface for the five-rule JSON shape. It is pure additive schema: zero risk to v1.3.3 shipping Friday, zero regression risk. The foundation stretch ends with T3 — the PolicyResolverService with merchant > tier > industry > global precedence. That resolver is the contract the Compliance Portal calls, the refactored CoaHelper calls, the scan pipeline calls, the license evaluator calls. Without it, every downstream ticket has no stable anchor. With it, the roadmap locks in.
What T1 → T3 unblocks
  • ✓ T2 — Policy CRUD service & controller (needs the tables to write into)
  • ✓ T3 — PolicyResolverService.getEffectivePolicy() — the contract every other ticket calls
  • ✓ T4 — CoaHelper rewrite (needs resolver to be real)
  • ✓ T5 · T6 · T7 — scan config, license evaluator, re-evaluate endpoint (all read through the resolver)
  • ✓ T8 — Admin UI policy authoring form (needs CRUD endpoints live)
  • ✓ Every subsequent MVP ticket. T1 → T3 is the root of the dependency tree.

What neither team will do

Five anti-patterns · guardrails for both sides

Anti-pattern · Scanner side
Building a parallel rules engine
The existing json-rules-engine stays unused. The registry holds definitions; existing evaluators run the logic. A second rules layer contradicts the 2026-04-12 policy-first pivot.
Anti-pattern · Policy side
Reaching into scanner internals
Policy rules reference check codes as opaque strings. Policy does not need to know if a check is KEYWORD vs DOCUMENT_PARSE — that is scanner-implementation detail. Publish validation confirms the code exists; nothing more.
Anti-pattern · Both sides
Duplicating the config store
One store for check config: pl_check_definition_version. One store for merchant overrides: pl_company_check_override. Policy overlays write to the shared override table. No parallel policy-only config table exists.
Anti-pattern · Timing
Merging the two MVPs now
Brian's spec is explicit: both ship in parallel. Phase 8 merges them — but because the check code has been the contract from day one, the merge is a connection, not a rewrite. Forcing an early merge only creates coordination overhead.
Anti-pattern · Scanner side
Skipping a check code
Every shipping check gets a code at seed. No exceptions. A check without a code today is a database rewrite in Phase 8. The S1 seed migration is the load-bearing Phase 1 artifact for exactly this reason.

The architectural payoff
Phase 1 ships two systems that already know how to talk to each other. Every later phase is strictly additive — a new rule type, a new executor, a new UI, a new source of config — never a rewrite of what was already shipped. That is the definition of a smart first step.
