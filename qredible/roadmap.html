Policy · Day 1
Policy Track · opens the week

Make "a compliance rule" a thing the platform can hold

Proposed tickets Pol-1 → Pol-3 · foundation stretch
① Business requirement
A compliance admin needs to be able to author a named compliance rule (e.g. "Cannabis Hemp Baseline") and say who it applies to (all merchants, an industry, a tier, or a specific merchant list). When a merchant matches more than one, the most specific one wins. Today the platform has no concept of a policy as a first-class object — rules are scattered across code and per-merchant toggles. The first step gives the platform a place to put a policy and a way to ask "which policy applies to this merchant right now?"
② What the admin will see
A new "Policies" tile on the existing Q-Audit → Global Settings grid (hypothesis — sits alongside Document Types, Industry Types, COA Field Library, Tiers). Tile click opens a Policy list page; list row click opens the authoring form. By end of Day 1–3 the list + tile are visible and a policy can be created from a form — publishing it has no downstream effect yet because the evaluator hasn't been wired.
Existing mockup: Policy Editor (full v3) Existing mockup: Policy Authoring Wizard MVP carve-out: 6 Claude Design surfaces pending
③ What it hooks into
Upstream: Q-Audit Global Settings (exists today — 17 tiles, a Policy tile already exists in that grid). Downstream: every subsequent policy ticket (CRUD, resolver, CoaHelper rewrite, scan config, license evaluator, re-evaluate button, admin UI form, case detail attribution). Without the foundation stretch, none of the downstream tickets have a stable anchor to call.
④ Recommended technical approach · guidance only
Two additive tables holding policies and merchant-to-policy assignments; a thin service that, given a merchant id, returns the one effective policy using merchant > tier > industry > global precedence with a publishedAt tiebreaker. Proposed names for shared coordination: pl_policy proposed and pl_merchant_policy proposed, wrapped by a PolicyResolverService proposed. The Policy team owns final names and shape. Rule payload stored as JSON (five rule types — COA expiration, Delta-9 max, THC-O max, license expiration, scan checks).
⑤ Acceptance — business-observable
  • ✓ Compliance admin can click into a Policies list and see 2–3 seeded example policies.
  • ✓ Admin opens "Cannabis Hemp Baseline v1", edits Delta-9 max %, saves — change persists.
  • ✓ Given a merchant id, the platform can answer: "What policy currently applies to this merchant?"
  • ✓ Zero user-facing regression on v1.3.3 features — policy data is dormant until the evaluator is wired later in the week.

Policy MVP — Policy team

Pol-1 → Pol-9

Policy Lifecycle MVP

Repos: RTQ-Admin-API, RTQ-Admin-UI
Per-phase format · business-first
① Business requirement
A compliance admin at Qredible needs to define what "compliant" means for a group of merchants — expressed as a named, versioned, scoped rule set — and change those rules without requiring a developer or a deploy. A merchant must receive a clear, attributable case when their COA, license, or site content falls outside the rules that apply to them. Today rules are hardcoded thresholds and per-merchant toggles scattered across the code; there is no durable object called "a policy." This MVP makes the platform understand a policy as a first-class thing.
② What the user will see
Primary user: Qredible compliance admin (single role for MVP — no approval gating, no author/approver split). Merchants see the downstream case in the existing Compliance Cases Portal.
  • Q-Audit → Global Settings gains a "Policies" tile — click it to reach the policy list (hypothesis: lives in the existing 17-tile grid next to Document Types, Industry Types, Tiers).
  • The policy list shows name, version, status, scope summary, published date. Row click opens the authoring form.
  • The policy authoring form has five enable-able rule cards (COA expiration, Delta-9 max, THC-O max, license expiration, scan checks) and a scope picker with four modes.
  • On the merchant detail page, a new "Re-evaluate policies" button in the actions area.
  • On a case detail page in the Compliance Cases Portal, one new row: Triggered by: [policy name] v[version] / [rule key] — linking back to the policy.
Policy Editor (full v3 — authoring surface) Policy Authoring Wizard (Noah's guided stepper) Multi-Policy Merchant Assignment MVP carve-out: 6 surfaces (policy-list-mvp, policy-edit-mvp, policy-scope-preview, merchant-reevaluate-trigger, case-detail-policy-stamp, q-audit-policies-tile) commissioned to Claude Design
③ What it hooks into
Upstream / existing screens this extends:
  • Q-Audit Global Settings — the 17-tile admin configuration grid (already has a "Policy" tile; this MVP fills it in).
  • Merchant detail page in Q-Monitor admin — gains the "Re-evaluate policies" action button.
  • Compliance Cases Portal — nothing built yet in code, but the outcome surface for this MVP lands here as policy-attributed findings once the portal is built.
Downstream / new surfaces this MVP adds: policy list, policy authoring form, scope preview modal, re-evaluate confirmation modal. Integration touch points (not user-visible): the existing CoaHelper logic that determines COA disposition, the scan pipeline that reads per-merchant scan config, the license expiration evaluator — all of these start reading from the published policy instead of from per-merchant toggles.
④ Recommended technical approach · guidance, not directive
The shape of the work (offered so the two tracks can coordinate; dev team chooses final names):
  • Two additive tables — one for policies (pl_policy proposed), one for merchant-to-policy assignments (pl_merchant_policy proposed).
  • A rules payload stored as JSON on the policy row (five rule keys: coaExpiration, delta9Thc, thcO, licenseExpiration, scanChecks).
  • A service that, given a merchant id, returns the one effective policy using merchant > tier > industry > global precedence with a publishedAt tiebreaker — proposed as PolicyResolverService proposed.
  • Existing rule-consuming code paths (CoaHelper, scan config, license evaluator) refactored to read through that resolver instead of per-merchant columns.
  • A new column on findings (policy_id proposed + policy_rule_key proposed) so cases can be attributed back.
What the dev team owns entirely: final entity names, file layouts, migration structure, TypeORM vs. raw SQL, caching strategy, JSON vs. JSONB, how to model the four scope modes.
⑤ Acceptance · business-observable

  • A merchant in the Hemp CBD industry uploads a COA showing 0.45% Delta-9. On the next evaluation (or via the admin's "Re-evaluate" button), a Compliance Case appears.
  • The case detail page shows "Triggered by: Cannabis Hemp Baseline v1 / delta9Thc" — hyperlinked back to the policy.
  • An admin can change the threshold to 0.2%, publish v2, re-evaluate that merchant, and see the case update to reference v2.
  • Zero regressions on existing COA disposition, scan, or license-expiration behavior for merchants not covered by a policy.

What neither team will do

Five anti-patterns · guardrails for both sides

Anti-pattern · Scanner side
Building a parallel rules engine
The existing json-rules-engine stays unused. The registry holds definitions; existing evaluators run the logic. A second rules layer contradicts the 2026-04-12 policy-first pivot.
Anti-pattern · Policy side
Reaching into scanner internals
Policy rules reference check codes as opaque strings. Policy does not need to know if a check is KEYWORD vs DOCUMENT_PARSE — that is scanner-implementation detail. Publish validation confirms the code exists; nothing more.
Anti-pattern · Both sides
Duplicating the config store
One store for check config: pl_check_definition_version. One store for merchant overrides: pl_company_check_override. Policy overlays write to the shared override table. No parallel policy-only config table exists.
Anti-pattern · Timing
Merging the two MVPs now
Brian's spec is explicit: both ship in parallel. Phase 8 merges them — but because the check code has been the contract from day one, the merge is a connection, not a rewrite. Forcing an early merge only creates coordination overhead.
Anti-pattern · Scanner side
Skipping a check code
Every shipping check gets a code at seed. No exceptions. A check without a code today is a database rewrite in Phase 8. The Scan-1 seed migration is the load-bearing Phase 1 artifact for exactly this reason.

The architectural payoff
Phase 1 ships two systems that already know how to talk to each other. Every later phase is strictly additive — a new rule type, a new executor, a new UI, a new source of config — never a rewrite of what was already shipped. That is the definition of a smart first step.
